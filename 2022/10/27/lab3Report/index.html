Deadlock Freedom

ZiFeng Zhao
Computer ArchitectureNoC Computer Architecture NoC

发布于：2022年10月27日

1. 实验目的

本实验在gem5 GARNET 3.0 模型中能够测量NoC是否存在死锁；以及通过Turn模型、Escape VC两种方式消除NoC中的死锁。

2. 实验原理

NoC产生死锁的根本原因是资源发生了循环依赖。对于以flit为传输单位的网络来说，这里的资源是指（虚拟）通道。死锁会造成传输的数据包永远无法到达目的节点。解决这一问题有两种思路：避免死锁；允许出现死锁但需要在发生死锁时有检测与恢复机制处理处于死锁状态的数据包。避免死锁一般也分两种思路：在路由算法层面消除死锁——如Turn模型；在流量控制层面消除死锁——如添加Escape VC，使数据包可以通过逃逸通道离开发生循环依赖的原通道。

3. 实验内容

3.1 死锁检测

分别对于以下三种Traffic Pattern: Uniform Random; Bit Complement; Tornado，确定以下两种路由算法：XY; random_oblivious的峰值吞吐率。

网络均采用8 * 8的Mesh网络。仿真周期为20000；每层虚拟网络有4个虚拟通道。XY算法即先X后Y. 这里的random_oblivious是指最小随机路由算法，即随机选择一条起始与终点节点之间的最小路径，而不考虑网络堵塞情况。例如(0,0)—>(2,1)，有(0,0)->(1,0)->(2,0)->(2,1); (0,0)->(1,0)->(1,1)->(2,1); (0,0)->(0,1)->(1,1)->(2,1)三种最短路径，该算法随机选择其中一种路径作为路由。显然这种路由算法会死锁出现。

定义：

  • 接收率：(packets/node/cycle)，每个周期每个节点所接收的平均数据包数量。
  • 峰值吞吐率：该网络配置提供的最大接收率。

在测量不同Traffic Pattern; 不同路由算法配置下的网络峰值吞吐率时，对于XY算法来说，由于不会发生死锁，随着注入率的增加，接收率会逐渐增加，最后在注入率的饱和点达到饱和，不再增加。为方便测量，可以直接设置注入率为1，此时必然饱和，测量网络所有节点接收到的数据包总数即可算出最大接收率，即峰值吞吐率。 而对于random_oblivious算法来说，由于会发生死锁，随着注入率的增加，接收率先会逐渐增加，当注入的包的数量到达一定程度后，网络中开始出现死锁，占用虚拟通道资源；继续增加注入率，死锁越来越多，最后虚拟通道资源基本全部被占用，之后继续仿真也不会有数据包被接收，接收率降为0. 因此对于会发生死锁的路由算法来说，近似认为最大接收率出现在恰好发生死锁之前的注入率点上。在gem5的模拟过程中，如果发生了死锁，会报错。因此可以逐渐增加注入率到恰好不发生死锁时，测量此时的接收率。

3.2 使用Turn model避免死锁

设定在GARNET中通过使用North-last Turn Model避免死锁。设定路由算法是random_oblivious. （路由算法实现在src/mem/ruby/network/garnet/RoutingUnit.cc中）。要求实现函数 outportComputeTurnModelOblivious() （提示：类似的可供参考的路由算法实现函数有outportComputeXY()和outportComputeRandomOblivious()

实现路由函数后可在命令行中指定–routing-algorithm=turn_model_oblivious设置路由算法

==补充题== ： 将路由算法从random_oblivious改为自适应最小路由算法，实现North-last Turn Model. 这里的自适应最小路由算法内容是：当路由器正在选择输出端口时，将所有最小路径上的下一个路由器的输入端的空闲VC数量作为优先级选择输出端口。如果存在多个相同最大输入VC数的情况，则在其中随机选择一个。（提示：可以参考has_free_vc()和select_free_vc()函数）

4. 实验过程

对于采用North-last Turn的自适应最短路径路由算法来说，如果只有一条最短路径，则无论该路径上的交通情况都选择该路径。如果有几条最短路径，例如目的节点在当前节点的东南方向，有两种最短路径：先南后东 以及 先东后南。 将这两条路径作为候选路径，进一步考察每个路径对应的输出端口上有多少个空闲的vc，选出空闲vc数最多的输出端口。

因此该算法分为两步：

step1: 选出所有可能的最短路径。设目的节点与当前节点的相对方向为relDir. 如果relDir = E, N, W, S，只有一条最短路径。 如果relDir = EN, North-last Turn规定只能先E后N, 如果relDir = NW, 只能先W后N, 如果relDir = WS, 有两条最短路径：先W后S或先S后W， 如果relDIr = SE, 有两条最短路径：先S后E或先E后S. step2: 如果relDir = WS/SE，计算每个输出端口的free vc数量进而选出相应输出端口。在OutputUnit.hh中添加 countFreeOutvc(int vnet, int outport) 函数用于计算给定网络、给定输出端口的free outvc数量。

相比于oblivious的minimal routing来说，adaptive算法需要注意的点是它需要考虑所计算的当前的数据包在哪层网络上，只有确定了数据包的网络位置，才能去看输出端口的free outvc数量。而数据包的网络位置信息可以从RouteInfo中获取。对于oblivious来说，它只用到了RouteInfo中的目的节点位置信息，而adaptive不仅需要利用该信息，还需要额外使用vnet（所在网络层）信息。

minimal adaptive routing with north-last turn算法在RoutingUnit.cc的outportComputeTurnModelAdaptive.

对于采用Escape VC的随机最短路径路由算法来说，题目设定了每层网络拥有4个虚拟通道，我们可以将vc0作为逃离通道，其他三个vc都采用oblivious minimal path routing算法。vc0作为逃离通道，要保证数据包进入vc0一定可以避免死锁最简单的方法是vc0中采用XY算法。我们需要考虑一个数据包应该何时进入逃离通道？可以在检测到死锁发生时（检测方法可以是设置一个死锁阈值，当一个数据包从源节点发出时开始计数，到达目的节点清0，数据包每次移动一个Hop，位于数据包中的计数器加1，如果超过死锁阈值，则认为死锁，如果此时这个数据包所在的节点的逃离通道空闲，则进入逃离通道，如果不空闲，则等到它空闲下来再进入逃离通道。）让数据包进入逃离通道。 也可以在每次allocate outvc时采用round-robin 方式，这样会使数据包可能还没发生死锁就提前进入逃离通道。当数据包进入逃离通道后，可以让其一直在逃离通道中传输，直到到达目的节点；也可以让其在之后重新回到其他三个通道中。直观角度看，前者的throughput会低于后者，因为1. XY算法没有Path Diversity，吞吐率低于oblivious算法。 2. 只有1个逃离通道，相比于3个普通通道，数据包进入逃离通道后更容易堵塞。而进入普通通道后，每次allocate outvc时有4个vc可供选择（包括逃离通道），因此直观上看因堵塞造成的延迟应该更低。

为了算法易于实现，我们设定每次采用一定通道优先级的方式分配4个Outvc（包括逃离通道）。(这也是gem5的 ==selct_free_vc== 函数所采用的outvc分配方式。该函数根据给定的网络层号，从小到大依次考察该网络层中的所有Outvc状态是否空闲，例如网络层0给定，依次考察vc0,vc1,vc2,vc3. 如果vc0不空闲，vc1空闲，则选择vc1. ) 当某条普通通道发生死锁后，数据包总可以在有限时间内进入其他空闲通道。在最坏情况中，该网络层的输出端口的三条普通通道全部发生死锁，逃离通道尽管可能堵塞，但总可以在有限时间之后重新空闲，因此数据包总是可以进入逃离通道。如果将0通道设为逃离通道，则每次会优先选择逃离通道，而逃离通道采用XY算法，吞吐率低于普通通道，因此可以将选择优先级最低的3号通道设为逃离通道. 这里需要做实验验证逃离通道的优先级不同是否会影响吞吐率。

修改：上面一段思路是设置一个采用XY算法的逃离通道，但这样有问题。首先介绍一下GARNET的Switch Allocator过程。

在GARNET模型中，Switch Allocator在第一阶段遍历考察该路由器的每个输入端口。根据当前考察的输入端口号，按照round robin的方式（round robin出来的最高优先级的vc由Switch Allocator上一次进行仲裁的第二阶段设定，设定方式是上一次Switch Allocator的仲裁第一阶段中所选择出来的round robin winner的下一个invc; 例如对于输入端口“East”来说，假设上一次Switch Allocator的仲裁第一阶段选择的invc是1号vc，则上一次Switch Allocator的仲裁第二阶段设定的本次具有最高优先级的是vc2. ）首先考察作为round robin 最高优先级的invc, 根据所考察的invc号，对给定的输入端口号和给定的invc号采用一定的==路由算法== ，获得输出端口号。然后根据计算出的输出端口号和上一次设置的outvc判断是否允许向该输出端口发送flit. 如果允许发送，则将该输出端口号设为当前所考察的输入端口号在第二阶段所发送flit的唯一输出端口号，将该invc号作为本轮仲裁的round robin winner. 然后结束对vc的考察。 如果不允许发送，则继续考察该invc的下一个invc. 如果当前考察的输入端口号的4个invc都没有赢得本轮仲裁，即它们中的flit都不能在Switch Allocator的下一阶段发送给相应的outvc，则会在下一次SA的第一轮仲裁阶段重新考察该输入端口号是否有invc可以赢得仲裁，发送flit. 考察完成一个输入端口后还会考察下一个输入端口，应该需要考察5个输入端口——东西南北和本地。显然经过第一轮仲裁后，可能出现两个不同的输入端口都允许向同一个输出端口发送flit的情况。这就需要第二阶段的仲裁。

Switch Allocator的第二阶段遍历考察该路由器的每个输出端口。根据当前考察的输出端口号，按照round robin的方式首先选出最高优先级的输入端口号，考察该输入端口是否允许向该输出端口发送flit. 如果不允许，考察下一个输入端口。如果允许，则通过判断该invc所对应的上一次的outvc是否是-1（如果是-1，说明上一次的flit是Head或Head_Tail，如果不是-1，说明上一次flit是Body）。如果上一次flit是body，本次flit不需要分配outvc，只需要沿着上一次outvc走即可。如果是 head或head_tail，则需要为本次flit在输出端口处分配一个空闲的outvc. （这里分配方式是按照outvc的编号有小到大的优先级分配的，例如outvc0空闲，则分配outvc0， 如果outvc0不空闲，Outvc1空闲，则分配outvc1. ）分配outvc后就可以向该outvc发送flit并减少该outvc的credit了。此外还需要向invc的Upstream路由器发送增加credit的信号。然后停止对当前的输出端口的考察，继续考察下一个输出端口。

回来原文，上面的思路问题在于所设置的逃离通道只采用了XY算法，而普通网络采用的是minimal oblivious算法。下面可能发生的场景会导致算法出错：一个数据包原先在普通通道中，在某个位于目的节点西南方向的节点中，根据minimal oblivious算法，有0.5的概率计算出来向北的输出端口。此时北输出端口恰好有空闲的outvc，于是它进入了Switch Allocator的第二个仲裁阶段。恰好第二个阶段将该数据包分配给了空闲的逃离通道，而此时目的节点在当前节点的东北方向，逃离通道采用的是XY算法，要求必须与目的节点的X坐标相同才可以转到Y方向，因此路由算法出错！

一种比较简单的解决方法是设置两个逃离通道，通道1采用XY算法，通道2采用YX算法。当数据包位于普通通道时，根据minimal oblivious算法，如果选择了东西端口，则可以选择普通通道或通道1（不能选择通道2，即使此时通道2是空闲也不能，一旦在东西端口进入通道2，则不能转到南北方向，在最坏情况中，例如一个flit的源节点是00，目的节点是21，在00选择东端口的通道2，根据YX算法，在10节点计算出来的输出端口一定是东端口，可能又恰好进入通道2到达20，由于仍采用YX算法，因此继续向东到达30，而这就违反了最小路径）；如果选择了南北端口，则可以选择普通通道或通道2. 当数据包位于通道1时，根据XY算法，如果选了南北端口，可以选择三种通道，无论选择哪种通道，下一次计算出的输出端口一定是南北方向（因为是最小路径）；如果选择了东西端口，则只能选择普通通道或通道1，不能选择通道2，一旦在东西端口进入通道2，则不能转到南北方向（因为通道2是YX算法，进入X方向后计算出的输出端口方向一定是东西方向）。 数据包位于通道2时，根据YX算法，如果选了东西端口，可以选择三种通道；如果选了南北端口，则只能选择普通通道或通道2.

我们设定vc0,vc1为普通通道，在选择空闲通道时具有更高的优先级，设定vc2为XY通道，设定vc3为YX通道。

==outportComputeEscapeVCOblivious==函数首先需要判断输入该路由器的flit位于哪个通道。为此，需要修改==route_compute==函数的输入项，该函数最终调用了outportComputeEscapeVCOblivious函数。在输入参数中增加当前输入到该路由器的flit的相对通道号relative_vc. 由于选择outvc是在Switch Allocator的第二阶段，因此需要在该阶段设置一个专门的select函数，输入参数为相对通道号，以及第一阶段获得的输出端口方向（东西南北），在该函数中根据这两个参数选择合适的空闲通道。而==outportComputeEscapeOblivious==函数与其他的route计算函数一样，只输出一个outport. 交给位于SA第二阶段的==select_free_vc_restrained==函数选择相应的outvc.

这个算法的==局限==在于：

​ // 在有些情况下，即使输出端口有空闲通道，也不允许将flit发送给这个空闲通道

​ // 情况汇总如下：

​ // 当数据包位于普通通道（invc0,invc1)时，如果输出端口是X方向，不允许将其输出到YX通道(outvc3)

​ // 当数据包位于普通通道（invc0,invc1)时，如果输出端口是Y方向，不允许将其输出到XY通道(outvc2)

​ // 当数据包位于XY通道（invc2)时，如果输出端口是X方向，不允许将其输出到YX通道(outvc3)

​ // 当数据包位于XY通道（invc2)时，如果输出端口是Y方向，可以输出到任意通道

​ // 当数据包位于YX通道（invc3)时，如果输出端口是X方向，可以输出到任意通道

​ // 当数据包位于YX通道（invc2)时，如果输出端口是Y方向，不允许将其输出到XY通道(outvc2)

简单来说，对于EscapeVC算法来说，在很多情况下，某些输出虚拟通道是对Switch Allocator屏蔽的。

之后这是一个改进方向。

5.2 避免死锁

上图是在Uniform Random Pattern下接受率 vs 注入率的曲线。XY算法的吞吐率最高，而ObliviousEscape的吞吐率最低。说明在随机交通流量下，采用逃离通道避免死锁的性能反而最差。

上图是Transpose Pattern下的接受率 vs 注入率曲线。可以看到，XY的吞吐率最低。Transpose模式即目的节点的xy坐标与源节点的xy坐标相反：dst_x = src_y; dst_y = src_x XY算法要求先X后Y，不具备路径多样性。而其他三种算法的路径多样性高，因此在Transpose这种源-目的节点固定的模式下吞吐率大。对于uniform random这种随机交通模式，路径多样性对于网络堵塞的影响降低，而Escape通道算法中有些空闲通道在某些时候需要对数据包屏蔽，降低了通道利用率，因此Escape算法的吞吐率反而最低。
